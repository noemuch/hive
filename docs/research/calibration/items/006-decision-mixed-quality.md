# re: idempotency keys

ok so the question is whether we put idempotency keys in the request body or in a header and there's more going on here than people think so let me just dump this.

body vs header — the surface question is "which is more idiomatic" and people are citing Stripe (header) vs various RPC systems (body) but that's not actually the important axis. the important axis is: does the key participate in request validation or not. if it participates — e.g. signed payloads, HMAC over the canonical request — then it MUST be in the body because headers aren't in the signature envelope for most of our clients (mobile SDK v3, python SDK, the legacy java client). if it doesn't participate, header is fine and arguably cleaner because it doesn't pollute the domain schema. we haven't decided yet whether idempotency keys are part of the signed envelope and THAT is the decision we should be making, not body-vs-header, body-vs-header is downstream of that.

second thing — ttl. nobody is talking about ttl. the idempotency cache has to have a ttl otherwise it grows without bound. stripe uses 24h. 24h is wrong for us because our longest retry window (the mobile offline queue) is 72h. if we pick 24h a mobile client that was offline for 48h will retry and get a fresh execution, which is exactly what idempotency is supposed to prevent. so we need at least 72h. but 72h at our current write rate (~3k/s of idempotent ops at peak) means the cache is ~780M entries. that's not redis-shaped, that's postgres-shaped, or a dedicated kv with ttl. people proposing "just use redis" haven't done the math.

third thing — the scoping question. idempotency key uniqueness has to be scoped to SOMETHING. stripe scopes to (merchant_id, key). we need to decide: scope to (tenant_id, key) or (tenant_id, user_id, key)? the stricter scope is safer (a rogue request from another user with a colliding key can't replay another user's op) but it means the same user across two devices can't use the same key — which breaks the mobile+desktop sync case where the same logical op might be initiated from two places. i lean (tenant, key) with a server-side fingerprint check: if the key matches but the fingerprint (hash of canonical body) doesn't, return 409 instead of the cached response. this catches accidental key collisions without penalizing legitimate cross-device use.

fourth — retry semantics when the first request is STILL IN FLIGHT. this is the case everyone forgets. client sends request A, network hiccup, client retries request A' with the same key. server is still processing A. what does the server do with A'? three options: (a) return 409 conflict, client has to wait and re-retry (bad UX, pushes retry logic to the client), (b) block A' until A finishes, then return A's response (harder to implement, needs a promise/future per key), (c) let both through and hope the downstream is also idempotent (this is what we do today and it's wrong). (b) is correct. (a) is the lazy version. we should do (b).

putting it all together: signed envelope decision first (this week), then key goes in body if signed / header if not, 72h ttl in postgres with (tenant_id, key) uniqueness, fingerprint check for collisions, single-flight blocking on in-flight requests. i can write this up properly but wanted to get the reasoning out before we make a body-vs-header call that's actually answering the wrong question.

i'm recommending: pause the current PR, make the signed envelope decision, then come back to this with the answer already determined by that decision. if we can't make the signed envelope decision this sprint, then provisionally put the key in the body (it's safer to move later from body to header than vice versa because it's backwards compatible) and proceed.
